# Session-cookie authentication backed by Postgres (not JWT)

Authentication uses an opaque session token carried in an **HttpOnly, Secure, SameSite=Lax cookie**; the server persists the session in a Postgres `sessions` table and resolves the account on every request with a sliding expiry. This is chosen over JWT because the deployment is a single SPA against a single API against a single Postgres — the statelessness JWT sells buys nothing here, and we already touch Postgres on every request, so one indexed primary-key lookup on a small `sessions` table is noise. The decisive win is revocation: logout, "log out everywhere", and password-change invalidation are a `DELETE`, which is exactly where JWT hurts (a signed access token stays valid until it expires). "JWT done right" — short-lived access tokens plus rotating refresh tokens persisted server-side for revocation — just reintroduces the same DB session state with more moving parts. Frontend and API are deployed under one registrable domain (`app.landforger.app` + `api.landforger.app`) specifically so `SameSite=Lax` applies and no CSRF token is required. Implementation uses **`alexedwards/scs` with its Postgres store**; account identity is carried in the session data. We accept scs storing the session token as the table key rather than a hash at rest — a battle-tested library is worth more at V1 than hand-owning security-sensitive code.

## Considered Options

- **JWT bearer tokens** — rejected for V1: revocation is the constant pain point, token storage forces an XSS-vs-CSRF tradeoff (localStorage vs cookie), and the statelessness that justifies it goes unused with one API and one database. Revisit if we ever expose a public/third-party API, add a mobile client that handles cookies poorly, or split the backend into independently-deployed services — JWT issuance can be added alongside sessions without removing them.
- **Roll our own session layer (~100–150 lines)** — genuinely small, and would let us hash the token at rest and use explicit columns (`user_id`, `user_agent`, `ip`). Deferred in favor of `scs`; reconsider if the hashed-token-at-rest requirement becomes firm or we outgrow scs's schema.
- **`gorilla/sessions`** — more manual, and its cookie-vs-server-store split is less aligned with the pure server-side model we want.
